# Bloom

## ビジョン

**適当な線を引くだけで、水彩画・水墨画のような「良い感じ」の絵になる 2D 描画ソフト。**

Photoshop や Illustrator のような本格的な 2D 描画を基本としつつ、「上手く描く技術」ではなく「描く気持ちよさ」を提供する。雑なストロークが画材の物理(滲み・かすれ)と適度な補正によって、それらしい絵に化ける体験がコア。

まずは自分用ツールとして作る。配布・他者向けの磨き込みは後回し。

### 「良い感じ」の定義(成功条件)

- 自分が雑に描いた線が、見て嬉しい水彩/水墨の表現になる
- 描いている最中の手応え(滲んでいく様子、筆のかすれ)が気持ちいい
- 「失敗した感」が出にくい — 補正と画材表現が下手さを吸収してくれる

## コア体験: ハイブリッドアプローチ

「適当な線から良い感じの絵」は 2 つのレイヤーの組み合わせで実現する。

### レイヤー A: 画材シミュレーション

ストローク自体が本物の水彩・墨のように振る舞う。

- **滲み(ウェット拡散)**: 紙の上の水分を流体シミュレーションで動かし、顔料が広がる
- **エッジの濃淡**: 乾燥時に縁へ顔料が溜まる水彩特有のエッジダークニング
- **粒状感・紙テクスチャ**: 紙の凹凸が顔料の溜まり方に影響する
- **かすれ**: 水墨画的なドライブラシ表現。筆の水分量・速度・筆圧で変化
- 技術的な定番は浅水方程式ベースの流体シミュレーション + Kubelka-Munk 顔料合成
  (古典: Curtis et al. "Computer-Generated Watercolor", SIGGRAPH 1997。商用の代表例: Rebelle)
- リアルタイムで動かすには GPU コンピュート(Metal compute shader)が前提

### レイヤー B: ストローク美化

描いた線そのものをアルゴリズムが整える。

- **手ブレ補正・スタビライザー**: 入力点列のスムージング
- **線の整形**: 入り・抜きの形状補正、筆圧カーブの調整
- **スタイライズ**(発展): 線の形状から筆致を再解釈して仕上げる

### 入力デバイス

- **ペンタブの筆圧検知は必須**。筆圧(+ できればチルト)がブラシの太さ・水分量・かすれに直結する
- **一次テスト機は手元の XPPEN Deco(板タブ)**。詳細は「入力ハードウェア」の節を参照
- **マウスでも良い感じに描ける**: カーソル速度から擬似筆圧を生成する(速く動かすと細く・かすれる)。筆圧前提の設計と両立できる確立した手法

## 機能スコープ

### MVP に入れるもの

- キャンバス(単一でも可)+ レイヤー
- 滲む水彩ブラシ・かすれる墨ブラシ(各 1 種でいい)
- 筆圧対応、マウス擬似筆圧
- 手ブレ補正
- undo/redo、画像書き出し(PNG)、ドキュメント保存

### 将来やるもの

- **3D 配置**: 手動で 3D モデリングするのではなく、2D で描いたものが 3D 空間に配置される(レイヤーが空間に浮かぶジオラマ/ビルボード的な表現)
- **MCP サーバ対応**: AI エージェントが Bloom のドキュメントを操作できる(ストロークを描く、レイヤーを操作する、書き出す等)
- ブラシの種類拡充、カラーパレット・混色 UI、アニメーション等

## 技術方針

### プラットフォーム比較

| 軸 | macOS ネイティブ (Swift + Metal) | Tauri (Rust + WebView) | Web / Electron |
|---|---|---|---|
| 筆圧・チルト | ◎ `NSEvent` に標準で筆圧・チルト・回転が乗る | △ WebView の Pointer Events 頼みで不安定 | ○ Pointer Events(Chromium は比較的良好) |
| GPU 計算 | ◎ Metal compute 直接 | ○ wgpu | ○ WebGPU |
| 入力レイテンシ | ◎ 最短 | ○ | △ |
| 将来の 3D | ◎ SceneKit / RealityKit / Metal | ○ | ○ Three.js 等 |
| MCP 統合 | ○ 公式 Swift SDK あり(若い) | ◎ Rust SDK 成熟 | ◎ TypeScript SDK 成熟 |
| 開発速度(自分のスキル) | ◎ Swift 経験あり | △ | △ |
| 配布のしやすさ | △ macOS のみ | ○ | ◎ |

### 結論: **Swift + Metal の macOS ネイティブで作る**

理由:

1. **自分用ツール**なので配布性(クロスプラットフォーム/Web)の優先度は低い
2. **Swift 経験がある** — 学習コストを画材シミュレーションという本丸に集中できる
3. このプロダクトの生命線である**筆圧入力の質と低レイテンシ**はネイティブが最も強い。Wacom 等のタブレットはドライバが標準 `NSEvent`(tablet subtype)として筆圧・チルトを届けてくれるため、特別な SDK なしで取れる
4. 滲みシミュレーションに必要な GPU コンピュートは Metal でそのまま書ける
5. 将来の 3D も Apple プラットフォームの 3D スタック(SceneKit/RealityKit)に自然につながる

弱点である MCP は、公式 Swift SDK(modelcontextprotocol/swift-sdk、2026 年初に再出発したばかりで若い)を使うか、stdio の JSON-RPC を薄く自前実装する。どちらでも成立する(下のアーキテクチャ参照)。

### 入力ハードウェア(一次テスト機: XPPEN Deco)

最初にテストするペンタブは手元の XPPEN Deco シリーズ(板タブ)。

**イベントの届き方**

- XPPEN は **ドライバ必須**。ドライバなしでは OS からただのマウスに見え、筆圧・チルトは取れない
- ドライバが動いていれば、Wacom と同様に標準の `NSEvent`(tablet イベント)として筆圧・チルトがアプリに届く(Krita / Photoshop 等が実際にこの経路で受けている)
- Wacom と違い **XPPEN は macOS 向け開発者 SDK を公開していない** → Bloom は標準 `NSEvent` の筆圧・チルトだけに依存する設計にする。これは Wacom でも同じ経路なので、結果的にベンダー非依存になる

**手元環境の確認済み事項(2026-06-08 時点)**

- ドライバはインストール済み: `/Applications/XPPen/XPPenTablet.app`(bundle ID `com.ugee.PenTablet`。UGEE は XPPEN の親会社)
- ただし現在タブレットは未接続・ドライバプロセスも停止中。設定もほぼ初期状態
- テスト前のチェックリスト:
  1. タブレットを USB 接続し、XPPenTablet.app を起動する
  2. macOS の「プライバシーとセキュリティ」で **アクセシビリティ + 入力監視**(必要なら画面収録)をドライバに許可する
  3. ドライバが古い場合は最新版(4.x 系)に更新する(3.x 系は Apple Silicon で不具合事例あり)

**注意点・リスク**

- **チルト対応はモデル依存**: Deco でも X3 ペン搭載世代はチルト対応、旧世代は非対応。接続時の入力プローブ(M0 参照)で実測して確認する
- **ドライバ品質に依存**: XPPEN の macOS ドライバは歴史的に不具合事例がある(権限不足で筆圧が死ぬ、OS アップデート後に動かない等)。筆圧が来ないときはまず権限とドライバ再起動を疑う
- **逃げ道**: ドライバ経由がどうしても不安定なら、`IOHIDManager` で HID を直接読む手がある(OpenTabletDriver が XPPEN のプロトコルをリバースエンジニアリング済みで参考にできる)。ただし大掛かりなので最終手段

## アーキテクチャ初期方針

後から変えにくい 2 点だけ、最初から織り込む。

### 1. UI と描画コアの分離

描画コア(ドキュメントモデル + ブラシエンジン + シミュレーション)は **UI なしで動くライブラリ**として作り、アプリはその上の薄い殻にする。

- コアへの操作は「コマンド」(ストロークを描く、レイヤー追加、書き出し等)として定義する
- → MCP サーバは同じコマンドを叩く薄いラッパーで済む
- → undo/redo もコマンド履歴として自然に実装できる
- → テストもヘッドレスで書ける

### 2. レイヤー = テクスチャ(平面)というドキュメントモデル

レイヤーを最初から「3D 空間に置ける平面(テクスチャ + 変換)」として持つ。MVP では全レイヤーが同一平面に並ぶだけだが、将来の 3D 配置はレイヤーに奥行きと姿勢を与えるだけの拡張になる。

## ロードマップ

- **M0 — 技術スパイク**: 2 段階でリスクを潰す
  - **M0a — 入力プローブ**: XPPEN Deco を繋ぎ、`NSEvent` の筆圧・チルト・proximity を画面にダンプするだけの最小アプリを作る。筆圧の分解能・イベントレート・チルトの有無を実測し、入力レイヤーの前提を固める(半日想定)— *実装済み・タブレットが戻り次第実測*
  - **M0b — 滲みブラシ**: Metal compute で滲みブラシ 1 本。ウィンドウに筆圧で描けて、水が広がり、乾く。ここが一番リスクが高い — ✅ **完了**([devlog 2026-06-08](devlog/2026-06-08-m0b-spike.md))
- **M1 — 最小ペイントアプリ**: キャンバス、レイヤー、undo/redo、PNG 書き出し、ドキュメント保存。コアと UI の分離をこの時点で確立する。UI は AppKit で素朴に作る(ネイティブなので Pencil 等の web/モバイル向けデザインツールは翻訳ギャップが大きく不採用)。レイアウトは簡単なスケッチで合意 → 実アプリ + スクショで反復、という流れ
  - 進捗([devlog](devlog/)): ✅ UI 枠(キャンバス + インスペクタ + ステータスバー)/ ✅ ブラシ設定 GUI / ✅ 任意色選択(RGB 顔料)/ ✅ レイヤー(複数・順序が効く合成・不透明度・表示切替・**D&D 並べ替え**)。⬜ undo/redo / ⬜ 保存・書き出し が残り
- **M2 — 美化系**: 手ブレ補正、マウス擬似筆圧、墨のかすれブラシ
- **M3 — MCP サーバ**: コアのコマンドを MCP ツールとして公開
- **M4 — 3D 配置**: レイヤーを 3D 空間に配置するビュー

## 未解決の問い・リスク

- **シミュレーションの性能**: 滲みをキャンバス全体でリアルタイムに回せるか。解像度・シミュレーショングリッドの粗さ・「濡れている領域だけ計算」等の工夫が要りそう。M0 で見極める
- **美化アルゴリズムの具体方式**: 手ブレ補正は確立しているが、「スタイライズ」をどこまでやるか(やりすぎると自分の絵でなくなる)
- **ファイルフォーマット**: 独自形式の中身(レイヤー構造 + ストローク履歴を持つか、ラスタのみか)。ストローク履歴を持つと再シミュレーションや MCP 操作と相性が良いが、複雑になる
- **乾燥の扱い**: 水彩の「乾くまで待つ/乾かす」を UI 体験としてどう設計するか(リアルタイム乾燥? 乾燥ボタン?)
- **Swift MCP SDK の若さ**: 公式 SDK が安定しなければ stdio JSON-RPC の自前実装に切り替える
- **XPPEN Deco のチルト対応**: 手元のモデルがチルト対応世代かは未確認。M0a の入力プローブで実測する。非対応なら当面は筆圧のみでブラシを設計する(チルトは「あれば良くなる」パラメータに留める)
